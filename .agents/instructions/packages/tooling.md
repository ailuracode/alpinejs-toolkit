# Tooling — Vitest + Vite

Normative rules for every Vitest and Vite configuration file in this
monorepo. Detailed rationale lives in
[`../skills/vitest-vite-tooling/SKILL.md`](../skills/vitest-vite-tooling/SKILL.md);
this file is the always-on rule set that applies whenever an agent edits
a Vitest or Vite config.

## Mandatory layout

| File                  | Lives in                    | Required by         |
| --------------------- | --------------------------- | ------------------- |
| `vitest.workspace.ts` | workspace root              | every package       |
| `vitest.config.ts`    | each `packages/*/` and root | every package, root |
| `vite.config.ts`      | root only                   | workspace demo app  |

The workspace file enumerates `vitest.config.ts` paths with a single glob
(`packages/*/vitest.config.ts`). Per-package overrides MUST live in the
package's own `vitest.config.ts`, never inside the workspace file.

`vite.config.ts` is **mandatory at the workspace root only**. Per-package
`vite.config.ts` is **opt-in** and only added when a package ships demo
content (an `index.html`, a `demo/` folder, or similar). For the toolkit
itself, demos are not published — `packages/*/vite.config.ts` does not
exist. Add one only when the package gains a real demo.

## Test file naming

Vitest file patterns MUST be:

```text
test/**\/*.spec.ts
test/**\/*.spec.tsx
test/**\/*.test.ts          # legacy, supported but discouraged
```

The legacy `*.test.ts` suffix is kept for compatibility with the original
`node --test` migration. New tests MUST use `*.spec.ts`.

## Test runner conventions

- `globals: true` is the default at the package level so `describe`,
  `it`, `expect`, `vi`, `beforeEach`, `afterEach`, etc. are auto-imported.
- `environment` MUST be set per package:
    - DOM-coupled packages (anything that exercises `matchMedia`,
      `IntersectionObserver`, `requestAnimationFrame`) MUST use `jsdom`.
    - Pure logic packages MAY use `node`.
- `setupFiles` MUST point to a `test/setup.ts` file inside the package
  when global stubs (matchMedia, fetch, ResizeObserver, etc.) are needed.
- `coverage` configuration MUST live in the workspace root, never per
  package. Each package sets only `include` for its own source files.

## Vite conventions

- Vite is used for the Astro demo app (`apps/demo/`) and for any package
  that ships its own demo content.
- Package `dist/` is generated by `tsc`, NEVER by `vite build`.
- The root `vite.config.ts` declares the `fmt` and `lint` blocks (see
  `.agents/instructions/shared/tooling-configs.md`). Format and lint
  rules live there.
- Per-package `vite.config.ts`, when it exists, MUST set `server.port`
  to a port that does NOT collide with the workspace host (`:5173` by
  default).
- `vite build` is reserved for shipping the demo app. Production
  library distribution remains the responsibility of `tsc`.

## When to add a `vite.config.ts` to a package

Add a `vite.config.ts` under `packages/<name>/` ONLY when the package
ships demo content (an `index.html`, a `demo/` folder, etc.). Examples:

- The package maintains a `demo/*.html` you can `vite preview`.
- The package is a UI component that needs a visual harness.

Do NOT add one for headless logic packages. A `vite.config.ts` with no
`index.html` produces a `404 Not Found` dev server — a dead config that
clutters the file tree and confuses future contributors.

## Forbidden patterns

- Reimplementing the Vite/Vitest config inline in package scripts.
- Running Vitest with the root config when the workspace file would pick
  up package overrides.
- Importing from `vitest` AND `node:test` in the same file.
- Using `describe.skip` / `it.skip` to silence failures. Fix the test
  or mark it `it.todo` with a tracker comment instead.
- Treating `vitest run` as `vitest watch` — CI MUST always use
  `vitest run`.
- Committing a `vite.config.ts` under `packages/*/` without demo content.

## Verification

Every PR that touches a Vitest or Vite config MUST run, locally:

```bash
bun run typecheck
bun run lint
bun test
bun run test:coverage
bun run build
bun --filter '@ailuracode/<name>' test
bun --filter '@ailuracode/<name>' test:coverage
bun --filter '@ailuracode/<name>' build
```

The closing-checklist in `AGENTS.md` references this file.
